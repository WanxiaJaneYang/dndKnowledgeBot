# 语料库摄入设计

> [English](../corpus_ingestion_design.md) | **中文**

## 1. 目的

本文档定义 D&D 3.5e 知识问答助手**语料库摄入层**的设计。

其目的是描述原始来源材料如何转变为**标准化、可追溯来源的规范语料库**，以供后续分块、索引和引用。

这是一份设计文档，而非实现文档。

## 2. 摄入的角色

摄入是将原始来源材料转换为结构化规范文档的过程。

其职责包括：

- 接受原始来源制品
- 提取可用内容
- 清理与标准化文本
- 保留来源溯源
- 为下游处理生成稳定的文档对象

摄入**不**负责：

- 最终检索质量调优
- 答案生成
- 引用渲染
- 确定最终分块策略

这些是下游关注点。

## 3. 为什么摄入是独立的层

项目应明确区分：

- 原始来源
- 规范文档
- 分块
- 已索引的检索单元

这种分离的重要性在于：

- 同一来源可能被多次重新分块
- 提取与标准化逻辑的变化方式与分块策略不同
- 即使分块发生变化，来源溯源也必须保持稳定
- 引用质量依赖于分块之前所保留的信息

因此，摄入层应生成可复用的规范语料库，而不仅仅是直接生成最终检索记录。

## 4. 第一阶段来源假设

第一阶段假设一个私人策划的 D&D 3.5e 语料库。

可能的来源类别包括：

- 用户以私人个人用途合法持有的核心规则书文本
- 有意纳入语料库的 SRD 对齐材料
- 未来的勘误或常见问题文档（作为单独标记的来源添加）

第一阶段排除不受控的外部材料，例如：

- 网络论坛
- 粉丝维基
- Reddit 帖子
- 同人规则集合
- 以 AI 生成摘要作为主要来源的内容

## 5. 摄入输入

摄入层应将每个来源视为明确追踪的制品。

潜在的输入形式包括：

- PDF 文件
- OCR 派生文本
- 手动整理的文本导出
- Markdown 或纯文本文档
- 后续创建的结构化来源记录

设计不应假设所有来源的清洁程度相同。

某些来源可能：

- 具有强结构信号
- 结构较弱但文本干净
- 包含 OCR 噪声
- 包含需要特殊处理的表格或侧边栏

## 6. 摄入输出

摄入的主要输出是**规范文档**。

规范文档应当：

- 标准化
- 结构化
- 可追溯来源
- 足够稳定以支持重新分块

规范文档尚不是向量索引记录，也不是最终答案单元。

## 7. 摄入流程

预期的摄入流程为：

```text
来源注册表条目
  -> 原始来源接入
  -> 提取
  -> 清理 / 标准化
  -> 结构映射
  -> 规范文档输出
```

每个步骤应保留或丰富来源溯源，而非丢弃。

## 8. 来源注册表依赖

摄入层依赖于来源注册表。

在摄入来源之前，它应具有至少包含以下内容的已追踪注册表条目：

- 来源标识符
- 来源标题
- 规则集 / 版本
- 来源类型
- 权威级别
- 状态
- 备注

来源注册表充当语料库准入的控制界面。

## 9. 原始来源接入

原始来源接入是将来源制品接受到项目中的环节。

其职责包括：

- 将物理或逻辑制品链接到已追踪的来源条目
- 在有用的情况下保留原始文件标识
- 记录摄入假设或注意事项

系统应避免将匿名文本块视为一等来源。

## 10. 提取

提取是将原始制品转换为可用文本与结构内容的步骤。

根据来源形式，提取可能涉及：

- 从数字文档中读取嵌入文本
- 消费 OCR 文本
- 读取结构化文本文件
- 在可获取时保留页面边界

提取应力求保留：

- 文本内容
- 页面位置
- 标题或章节标记
- 明显的列表或表格边界（如可获取）

提取不应对检索分块大小做出不可逆的假设。

## 11. 清理与标准化

提取后，内容应被清理为一致的形式。

标准化目标包括：

- 在安全的情况下移除重复的页眉页脚
- 纠正明显的提取噪声
- 标准化空白字符与换行
- 保留有意义的格式边界
- 避免意外丢失章节含义

系统应优先采用**最小损失标准化**，而非激进改写。

如果某个清理步骤有损害来源溯源或结构的风险，应谨慎对待。

## 12. 结构映射

结构映射是识别可用文档结构的步骤。

有用的结构可能包括：

- 书籍级别的标识
- 章节或小节层级
- 条目边界
- 页码范围
- 列表块
- 表格块
- 示例或侧边栏

目标不是完全解读每条规则，而是为下游分块与引用保留足够的结构。

## 13. 规范文档模型

规范文档应表示来源或来源片段的标准化视图。

从概念层面，规范文档应保留：

- 规范文档标识符
- 来源标识符
- 来源标题
- 版本 / 规则集
- 来源类型
- 文档标题（如适用）
- 章节路径（如适用）
- 页码范围
- 清理后的内容块
- 摄入版本或溯源元数据

确切的模式定义属于单独的模式文件，但设计要求是规范文档应稳定且可复用。

## 14. 来源溯源要求

来源溯源是强制性的。

摄入至少应保留足够的信息，以便后续支持：

- 来源标题引用
- 版本范围限定
- 基于页面的参考（如可获取）
- 基于章节路径的参考（如可获取）
- 从分块追溯到来源的可追溯性

如果摄入过程中丢失了来源溯源，后续的引用质量将会下降。

## 15. 版本控制与溯源

设计应假设摄入可能随时间重复进行。

原因包括：

- 提取质量改进
- 修订的标准化规则
- 来源更正
- 新的来源版本
- 更好的结构映射

因此，摄入模型应保留以下溯源概念：

- 来源版本
- 摄入版本
- 规范文档版本
- 已知提取问题的备注

这目前不需要最终的技术实现，但应成为概念模型的一部分。

## 16. 表格、列表与侧边材料的处理

摄入应保留特殊结构的存在，即使后续分块决定如何使用它们。

示例包括：

- 职业表格
- 装备表格
- 法术列表
- 项目符号列表
- 侧边栏
- 示例

摄入层不需要独立解决最终的表格检索问题，但应避免在可恢复结构存在的情况下将所有内容压平为非结构化文本。

## 17. 勘误与常见问题材料的处理

未来的勘误与常见问题文档应被视为独立的来源类型，而非悄然合并到旧文本中。

重要原因：

- 用户可能希望分别查阅原始文本与后续澄清
- 后续的纠正层可能需要明确的冲突策略
- 引用应使覆盖或澄清的来源可见

第一阶段不需要完整的勘误覆盖系统，但摄入设计应为此留有余地。

## 18. 准入策略

并非每个文本制品都应自动进入语料库。

来源应仅在以下情况下被准入：

- 经过有意选择
- 属于第一阶段的 D&D 3.5e 范围之内
- 来源溯源已知
- 适合在本项目中以私人个人用途使用

本项目应优先选择规模更小、质量更高的语料库，而非规模庞大但不受控的语料库。

## 19. 需要防范的失败模式

摄入设计应明确防范：

- 未知的来源标识
- 缺失的版本元数据
- 页面映射中断
- 过于激进的清理破坏了结构
- 意外合并不同的来源层
- 隐性的来源溯源丢失
- 将下游分块决策视为摄入事实

## 20. 摄入质量目标

一个好的摄入结果应满足以下大部分条件：

- 来源被清晰识别
- 文本可读且已标准化
- 页面映射尽可能保留
- 章节结构部分可恢复
- 来源溯源完整
- 规范输出可用于多种未来分块策略

## 21. 推迟的决策

以下问题有意推迟：

- 将使用什么确切的解析器或提取器？
- OCR 纠正将是手动、自动还是混合方式？
- 应存在多少种规范文档粒度？
- 表格应如何在结构上表示？
- 一个章节应映射到一个规范文档，还是粒度应更小？

这些属于后续实现或模式细化工作的范畴。

## 22. 摘要

一句话总结：

> 摄入层将精心策划的 D&D 3.5e 来源制品转换为标准化的规范语料库，保留来源溯源、文档结构与可复用性，以供后续分块与引用。
