# 架构概览

> [English](../architecture_overview.md) | **中文**

## 1. 目的

本文档定义 D&D 3.5e 知识问答助手的**高层架构**。

本文档有意**以设计为导向**，而非以实现为导向。目标是在选择具体框架、数据库或部署基础设施之前，明确模块边界、数据流向与系统职责。

## 2. 架构意图

系统旨在作为**有据可查的规则助手**运作。

从高层次来看，系统应：

1. 摄入精心策划的 D&D 3.5e 来源材料
2. 将其标准化为规范语料库
3. 拆分为可检索、可引用的证据分块
4. 从这些分块派生出供检索使用的索引产物
5. 从检索到的证据回答用户问题
6. 附上指向已保留来源溯源的引用

架构应优先考虑：

- 溯源优先于便利
- 明确的模块边界
- 稳定的中间表示
- 严格的规则集范围限定
- 弃答优先于无据流畅

## 3. 系统形态

预期系统形态为：

```text
精心策划的来源
  -> 摄入
  -> 规范语料库
  -> 分块
  -> 索引
  -> 检索
  -> 答案合成
  -> 引用渲染
  -> 用户响应
```

该架构将**文档准备**与**运行时问答**分离。

这种分离的原因：

- 来源处理的变化速度慢于检索策略
- 分块策略可能随时间演变
- 引用质量依赖于早期阶段保留的来源溯源
- 答案生成应消费精心策划的证据，而非原始文档

## 4. 核心模块

### 4.1 来源注册表

来源注册表追踪语料库清单。

其职责包括：

- 确定哪些来源在范围之内
- 分配稳定的来源标识符
- 追踪来源类型与版本
- 记录来源状态与备注
- 区分主要来源与未来的补充来源

来源注册表是**语料库控制层**，而非检索层。

### 4.2 摄入

摄入模块将原始来源制品转换为标准化的规范文档。

其职责包括：

- 读取来源材料
- 提取可用文本与结构
- 清理重复页眉页脚或 OCR 噪声等杂质
- 保留页面与章节的来源溯源
- 生成规范文档对象

摄入不应与单一分块策略紧密耦合。

### 4.3 规范语料库

规范语料库是原始来源与分块检索单元之间的稳定中间层。

其作用是提供：

- 标准化的文档表示
- 章节路径与来源原生 locator 等结构化元数据
- 稳定的来源溯源
- 可用于重新分块或重新索引的可复用基础

该层的存在使得分块策略可以演变，而无需从来源文件重新提取全部内容。

### 4.4 分块

分块模块将规范文档转换为可供检索与引用的稳定证据对象。

其职责包括：

- 保持语义连贯性
- 尽可能尊重规则边界
- 为每个分块保留来源溯源
- 分配分块级别的元数据
- 支持能够感知章节、条目与表格的分块策略

分块层是检索质量与引用质量的核心决定因素。

### 4.5 索引

索引层为运行时检索准备分块。

其职责包括：

- 将分块与嵌入或其他检索表示相关联
- 存储用于过滤的分块元数据
- 使分块可被后续检索逻辑搜索

索引产物是从分块派生的。它们应可在不重新定义稳定分块契约的前提下被替换。

该层应支持嵌入或混合检索策略的未来变更，而不会使规范语料库模型失效。

### 4.6 检索

检索层为用户问题选择支撑证据。

其职责包括：

- 理解用户查询
- 应用规则集与来源过滤器
- 检索可能相关的分块
- 将证据候选项返回给答案合成

在第一阶段，检索应假设**严格的 D&D 3.5e 语料库边界**。

第一阶段的实现为 lexical-first、domain-aware。具体流水线为：

1. **归一化** 原始查询 —— 大小写归一、标点清理、术语别名扩展、受保护短语（`scripts/retrieval/query_normalization.py`）。
2. **词法检索** 自 chunk 语料的 FTS/BM25 索引（`scripts/retrieval/lexical_index.py`）。
3. **命中信号水合** —— 精确短语、受保护短语、section-path 命中、词元重叠（`scripts/retrieval/match_signals.py`）。
4. **综合打分** —— BM25 + 命中信号加分 + chunk-type 先验（规则类 chunk 优先于示例类）。
5. **约束过滤** —— 在候选接受前应用 edition、source type、authority level 与已排除 source 的硬过滤。
6. **章节感知 shaping** —— 按 `(document_id, section_root)` 分组，按组内最佳排名排序（`scripts/retrieval/candidate_shaping.py`）。
7. **证据包** —— 将 shaping 后的输出与归一化查询、约束摘要、逐项证据（内容、chunk 类型、source_ref、locator、命中信号、section_root）以及用于调试的 pipeline trace 打包（`scripts/retrieval/evidence_pack.py`）。

语义/向量检索推迟至 `docs/chunking_retrieval_design.md` 所述的第二阶段 hybrid 扩展。

### 4.6a 答案阶段（基于规则，v1）

v1 答案阶段是叠加在检索证据包之上的基于规则的摘要式组合器。它将 `EvidencePack` 转换为有据答案（主摘要 + 最多两段辅助摘要，每段附带分块级引用）或弃答——不生成任何合成散文。这是有意为之的验证脚手架；后续 v2 将以同一 `EvidencePack` 契约引入基于 LLM 的散文合成器。

- 基于规则的摘要组合器——段落文本始终取自分块摘录，绝不生成
- 严格信号弃答门——首位证据项必须命中 exact_phrase_hit、protected_phrase_hit 或 section_path_hit 中至少一项
- 每条答案最多三段——一段主答 + 最多两段辅助（先同组兄弟，再取信号不同的跨章节项，槽位 2 回退为下一个兄弟）
- 引用绑定器——分配稳定的 citation_id，并对共享 `(chunk_id, excerpt)` 的段落去重
- 双开关 CLI——`--json` 输出严格 schema 形态；`--json-debug` 追加顶层 `debug` 块（非 schema 合规）携带流水线追踪与逐项 match_signals
- v1 验证脚手架——与计划中的 v2 LLM 组合器共享 `build_answer(pack) → AnswerResult` 接口

完整设计见 `docs/plans/2026-04-23-issue-23-minimal-answer-path.md`。

### 4.7 答案合成

答案合成层将检索到的证据转换为有据可查的响应。

其职责包括：

- 从检索到的证据回答用户问题
- 区分直接支撑与受支撑的推断
- 避免无据断言
- 在证据薄弱或相互矛盾时弃答
- 生成可携带引用的响应结构

答案层不应凭空捏造检索证据集中不存在的来源引用。

### 4.8 引用渲染

引用层将来源溯源格式化为用户可见的参考信息。

其职责包括：

- 将答案引用映射到来源元数据
- 渲染来源标题与位置信息
- 保持从答案到来源的可追溯性

引用不是表面的后处理步骤，它是跨越早期各层所保留的来源溯源在输出端的可见呈现。

## 5. 数据生命周期

系统采用分阶段的数据生命周期。

### 阶段 A — 原始来源层

示例：

- PDF 规则书
- 文本导出
- 结构化来源文本
- 未来的勘误文档

该层是原始输入，应与下游处理形式保持区分。

### 阶段 B — 规范文档层

该层包含从原始来源派生的标准化文档对象。

规范文档应保留：

- 来源标识
- 版本
- 文档结构
- 来源原生 locator 信息（如可获取）
- 清理后的内容

### 阶段 C — 分块层

该层包含从规范文档派生的稳定证据对象。

分块应保留：

- 分块标识
- source_ref
- 父规范文档标识
- locator
- 分块类型
- 分块文本

分块对象应保持为稳定的证据对象。嵌入向量或引擎特定的索引字段应归属于派生的索引层，而不应进入分块契约本身。

### 阶段 D — 索引产物层

该层包含按分块标识键控的派生检索产物。

示例包括：

- 嵌入向量
- ANN 索引元数据
- BM25 或混合检索产物
- 引擎特定的过滤记录

当检索策略发生变化时，该层应可低成本地重新生成。

### 阶段 E — 检索结果层

该层包含为用户查询选定为证据的分块子集。

检索结果应被视为**候选证据**，而非已可信任的最终断言。

### 阶段 F — 答案层

该层包含生成的有据响应与附带的引用。

答案层必须可追溯至检索结果集。

## 6. 运行时路径与准备路径

系统有两条主要流程。

### 6.1 准备路径

这是速度较慢的语料库构建路径。

```text
来源注册表
  -> 原始来源接入
  -> 摄入
  -> 规范语料库
  -> 分块
  -> 索引
```

该路径是建立语料库质量、来源溯源与分块质量的环节。

### 6.2 运行时问答路径

这是面向用户的路径。

```text
用户问题
  -> 查询理解
  -> 检索
  -> 证据集
  -> 答案合成
  -> 引用渲染
  -> 最终响应
```

该路径应依赖检索到的证据，而非模型的潜在记忆。

## 7. 设计原则

### 7.1 规范先于分块

架构应始终在生成分块之前保留规范语料库层。

### 7.2 默认情况下检索单元与引用单元对齐

默认地，分块应同时作为：

- 检索单元
- 默认引用锚点

这使得来源溯源保持稳定，并减少了答案生成时的歧义。

但这是设计上的默认，而非硬不变量。一个被检索到的分块后续可能产出多个更细粒度的引用 locator，多个分块也可能共同支撑同一段答案。

### 7.3 来源溯源从不可选

每个下游对象都应可追溯至某个来源。

### 7.4 版本边界是结构性的，而非建议性的

版本范围限定应编码在架构与元数据模型中，而不仅仅依赖提示词措辞。

### 7.5 回答受证据约束

答案层的存在是为了综合检索到的证据，而非取代检索。

## 8. 第一阶段假设

第一阶段架构假设如下：

- 私人单用户使用
- 仅限 D&D 3.5e
- 精心策划的语料库，而非开放式网络检索
- 设计优先的仓库状态
- 引用作为一等要求
- 在锁定基础设施之前先进行薄垂直切片验证

这些假设降低了复杂性，有助于防止过早泛化。

## 9. 第零阶段排序

第零阶段应在锁定基础设施相关决策之前，先验证一个薄的垂直切片。

这意味着：

- 选择一个真实的已准入来源切片，例如 SRD 内容或 PHB 的一个章节
- 针对该切片构建一个小型金标问题集
- 在经受真实的摄入、分块与引用示例考验之前，保持 schema 轻薄与暂定

过早地选择最终向量服务、最终模型集合或完全展开的 schema，会增加契约漂移风险。

## 10. 第一阶段排除的架构关注点

以下关注点有意推迟：

- 公开多用户系统设计
- 认证与多租户
- 多用户实时同步
- 生产部署拓扑
- 扩展架构
- 插件生态系统或第三方扩展
- 对多游戏系统的通用支持

## 11. 关键架构边界

### 边界 A — 来源边界
只有有意选择的 D&D 3.5e 语料库来源被视为在范围之内。

### 边界 B — 规范边界
原始文档与规范文档是不同的对象，不得混淆。

### 边界 C — 分块边界
分块处于规范标准化的下游，可独立演变。

### 边界 D — 索引边界
索引产物是从分块派生的，可在不重写分块语料的前提下重新生成。

### 边界 E — 证据边界
检索到的分块是答案层可用的证据集。

### 边界 F — 引用边界
用户可见的引用必须解析为已保留的来源溯源，而非模型自由输出。

## 12. 未来扩展点

架构应为未来的添加留有余地，而不强制将其纳入第一阶段。

潜在扩展点包括：

- 勘误与常见问题覆盖层
- 重排序
- 跨规则集或版本的对比模式
- 支持表格感知的结构化检索
- 来源优先级策略
- 引用提取策略

这些应有意添加，而非默认假设。

## 13. 待解设计问题

以下问题在薄垂直切片验证完成后仍未解决：

- 在初始切片之外，还有哪些分块类型值得追加？
- 引用 locator 何时应当比分块 locator 更细？
- 主文本与后续勘误之间的冲突应如何表示？
- 什么评测阈值足以支撑语料扩展？
- 哪些本地索引与模型选择对首次实现而言是足够好的？

## 14. 摘要

一句话总结：

> 本架构是一个分阶段、保留溯源的 RAG 流水线，将精心策划的 D&D 3.5e 来源材料转化为带有可追溯引用的有据答案。
