# D&D 3.5 知识问答助手

> [English](README.md) | **中文**

一个私人 RAG 项目，用于就**龙与地下城 3.5e** 规则提出有据可查的问题，并获得带有**来源引用**的答案。

## 项目状态

本仓库目前处于**设计阶段**。

当前阶段的重点是：

- 定义产品范围
- 定义语料库与来源边界
- 定义摄入、分块、检索与引用策略
- 定义评估标准

本仓库**尚未进入实现阶段**。代码、基础设施与部署方案应跟随设计文档，而非领先于设计文档。

## 愿景

目标是构建一个 D&D 规则助手，能够：

- 通过检索规则原文（而非依赖模型记忆）回答问题
- 为每个答案引用支撑来源
- 保持在清晰定义的规则集边界之内
- 在证据不足或存在歧义时弃答

第一阶段，助手仅限 **D&D 3.5e**。

## 为什么做这个项目

通用聊天模型在回答桌游规则问题时，往往存在以下问题：

- 记忆不完整
- 版本混淆
- 凭空确信
- 缺乏引用

本项目旨在通过将聊天机器人定位为**有据可查的规则参考助手**（而非通用奇幻聊天机器人）来解决上述问题。

## 核心原则

### 1. 有据胜于流畅
答案必须有检索到的证据支撑。一个不那么流畅但有据可查的答案，好过一个表述优美但凭空捏造的答案。

### 2. 引用是核心功能
引用不是 UI 附加项。检索与分块设计必须保留足够的来源溯源信息，以支持可信的引用。

### 3. 版本边界严格执行
第一阶段仅支持 **D&D 3.5e**。除非未来明确设计了跨版本对比模式，否则系统不应混用其他版本的内容。

### 4. 私人个人用途
本项目供私人个人使用，目前不设计为公开机器人、共享规则平台或商业产品。

### 5. 设计先于实现
架构、语料库策略与数据约定，优先于框架或部署方案的选择。

## 第一阶段范围

第一阶段范围有意保持精简。

已包含：

- 单用户私人使用
- D&D 3.5e 规则问答
- 来源感知的摄入设计
- 针对规则书与结构化条目的分块策略
- 有据回答的检索设计
- 带引用的答案格式
- 证据不足时的弃答行为

暂未包含：

- 多版本支持
- 同人或粉丝内容
- 战役记忆
- 遭遇模拟
- 角色构建优化引擎
- 公开部署
- 协作或多用户功能

## 初始产品形态

从高层次来看，预期产品形态为：

1. 将来源文档摄入规范语料库
2. 将规范文档拆分为可引用的检索单元
3. 将这些单元索引至向量检索层
4. 仅从检索到的证据回答用户问题
5. 返回带有来源引用的有据答案

## 来源策略

本系统旨在基于精心策划的语料库，而非开放式网络搜索。

初始来源优先级：

- `srd_35`，作为启动阶段已经准入的来源切片
- 用户合法持有、供个人私用的官方 D&D 3.5e 规则文本
- 有意纳入语料库的结构化补充来源材料
- 未来如添加勘误或常见问题文档，将作为单独标记的来源

系统应始终保持来源溯源。

对于启动阶段的 `srd_35`，仓库现在通过
`configs/bootstrap_sources/srd_35.manifest.json` 固定来源契约。
在仓库根目录运行 `python scripts/fetch_srd_35.py`，
即可在 `data/raw/srd_35/` 生成本地来源快照，同时保持原始 SRD 文件不进入 git。

## 回答策略

助手应：

- 基于检索到的证据回答
- 区分直接证据与推断
- 在语料库不支持时，避免假装确定
- 在答案层面引用支撑来源
- 在证据不足时明确说明

## 仓库理念

本仓库应保持精简、清晰、以设计为驱动。

推荐做法：

- 清晰的文档
- 显式的模式定义
- 明确的假设
- 阶段性的精简目标

应避免：

- 过早的框架决策
- 在范围稳定之前进行大量实现脚手架搭建
- 在同一文件中混合设计讨论与代码实验
